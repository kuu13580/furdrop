# 要件定義書 - FurDrop

## 1. サービス概要

不特定多数の撮影者（送信者）から、写真を受け取るためのサービス。
受信者が公開URLを発行し、送信者がそのURLに匿名で写真をアップロードする。

**ユースケース例:**
- イベント・ライブの参加者に「撮った写真をここに送ってください」と案内
- 名刺やSNSプロフィールにURLを掲載し、継続的に写真を受け付ける

### 1.1 他サービスとの差別化

既存の写真共有・ストレージサービス（Google Drive, Dropbox, AirDrop等）との差別化ポイント:

| ポイント | 説明 | 既存サービスの課題 |
|---|---|---|
| **匿名送信に特化** | 送信者・受信者ともにメールアドレス等のプライベート情報を交換せずに写真を送受信できる。URLを知っていれば誰でも送信可能 | Google Drive等は送信者にアカウント登録が必要。AirDropは物理的近接が必要 |
| **送信者情報の自動埋め込み** | 透かしやEXIFへの情報埋め込みにより、写真ファイル自体に送信者情報を残せる。どの方式を許可するかは受信者が制御（R14）。匿名性と追跡可能性を両立しつつ、受信者の意図しない加工を防止 | 匿名送信では「誰が送った写真か」の管理が困難。他サービスは送信者/受信者どちらの立場も考慮した制御がない |
| **写真受信に特化したUX** | 送信者名・日付でのソート/グルーピング、一括DL、ストレージ管理など、「大量の写真を受け取って管理する」ワークフローに最適化 | 汎用ストレージは写真受信に特化した管理機能がない |
| **DL期限による自動削除** | 受信した写真に閲覧期限を設け、期限後に自動削除する。無料のストレージサービスとして悪用されることを防止 | 無制限保存が前提のサービスが多く、コスト管理が困難 |
| **受信通知** | 写真を受信したらプッシュ通知で即座に知らせる。イベント中のリアルタイム受信に対応 | メール共有は通知が埋もれやすい |
| **セキュリティ（画像専用）** | サーバーサイドで画像フォーマットを検証し、画像以外のファイルをブロック。悪意あるファイルの送信を防止 | 汎用ファイル共有は任意のファイルをアップロード可能 |

---

## 2. ユーザー種別

| 種別 | 説明 | 認証 |
|---|---|---|
| 受信者 | 写真を受け取る人。アカウント登録が必要 | Firebase Auth（Twitter OAuth等） |
| 送信者 | 写真を送る人。匿名でURLにアクセス | 不要 |

## 3. 機能要件

### 3.1 受信者機能（認証必須）

| ID | 機能 | 優先度 | 説明 |
|---|---|---|---|
| R01 | アカウント登録 | MUST | Twitter OAuth等でログインし、公開URL用スラッグを設定 |
| R02 | 公開URLの発行・共有 | MUST | `https://domain/:slug` 形式。QRコード表示・コピー機能 |
| R03 | 受信写真の一覧表示 | MUST | サムネイルグリッド。ページネーション（無限スクロール） |
| R03a | ソート・グルーピング | WANT | 日付別・送信者別でのソート切替、セクション見出しによるグルーピング表示。写真選択・DL時の視認性向上 |
| R04 | 写真の詳細表示 | MUST | メタデータ（送信者名、ファイル情報）表示 |
| R05 | オリジナル画像のDL | MUST | 単体DL。署名付きURLで直接DL |
| R06 | 写真の削除 | MUST | 単体・一括削除。ストレージ使用量を即時反映 |
| R07 | ストレージ使用状況表示 | MUST | 使用量/上限をプログレスバーで表示 |
| R08 | 一括ダウンロード | WANT | 複数選択→個別DL URL発行（ZIP生成はWorkers制約上困難） |
| R09 | プッシュ通知（新着写真） | MUST | FCM経由。設定画面でON/OFF。写真受信をリアルタイムに把握するために必須 |
| R10 | プロフィール編集 | WANT | 表示名・スラッグ変更 |
| R11 | 受付停止/再開 | WANT | 一時的にアップロードを停止する機能 |
| R12 | フィードバック・不具合報告 | MUST | フッターにGoogleフォームへのリンク（メール収集OFF） |
| R13 | DL期限・自動削除 | MUST | 受信した写真に閲覧期限を設ける（デフォルト30日）。期限後は自動削除される。無料ストレージとしての悪用防止。ギャラリーで写真を閲覧開始した時点からカウント、または受信日からの固定期限（要検討） |
| R14 | 受信オプション設定 | MUST | 送信者に提示するオプションを受信者が制御できる。設定画面で切替 |

**R14 受信オプション設定の詳細:**

送信者情報の埋め込み方式は画像への影響度が異なるため、受信者が許可するオプションを選択できる。

| オプション | デフォルト | 画像への影響 | 受信者が制御する理由 |
|---|---|---|---|
| 送信者名入力 (S03) | ON | なし（DB保存のみ） | 基本機能として常時ON |
| EXIF埋め込み (S04) | OFF | メタデータのみ（除去可能） | 受信者が希望する場合のみ有効化 |
| 透かし (S05) | OFF | **不可逆**（画像を直接加工） | 受信者の意図しない透かしが入ると元に戻せない。別途クレジット表示したい場合等に不要 |

送信者のアップロード画面では、受信者が許可したオプションのみ表示される。

### 3.2 送信者機能（認証不要）

| ID | 機能 | 優先度 | 説明 |
|---|---|---|---|
| S01 | 受信者のランディングページ表示 | MUST | 受信者名・プロフィール表示。受付状態確認 |
| S02 | 複数枚写真アップロード | MUST | ドラッグ&ドロップ + ファイルピッカー。JPEG/PNG/HEIC対応（クライアントでJPEGに変換） |
| S03 | 送信者名の入力 | MUST | 任意。TwitterID等を入力可能。DBに保存され受信者に表示 |
| S04 | EXIF送信者情報埋め込み | WANT | 送信者が自由入力したテキストをEXIFのカメラモデルフィールドに書き込む。元のカメラモデル情報は上書きされる。画像ファイル自体に送信者情報を残す手段（S03のDB保存、S05の透かしとは別のアプローチ） |
| S05 | 透かし（ウォーターマーク）埋込 | WANT | クライアントサイドでCanvas描画。位置・サイズ・透明度を選択可能。画像に視覚的に送信者情報を残す手段 |
| S06 | アップロード進捗表示 | MUST | ファイルごとの進捗バー |
| S07 | 送信完了画面 | MUST | 完了メッセージ + 送信した写真のサムネイル一覧 |

### 3.3 システム機能

| ID | 機能 | 優先度 | 説明 |
|---|---|---|---|
| X01 | サムネイル生成 | MUST | クライアントサイド。長辺400px, JPEG品質0.7 |
| X02 | ストレージクォータ管理 | MUST | 受信者ごとの総容量制限。アップロード時に二重チェック |
| X03 | アップロードセッション管理 | MUST | 複数枚を1セッションにまとめる。1セッション最大100枚。1時間で期限切れ |
| X04 | 期限切れセッション・失敗写真のクリーンアップ | MUST | Cron Triggerで毎時実行 |
| X05 | レート制限 | WANT | 送信者のIPベース。セッション作成に対して制限 |
| X06 | 画像サイズ制限 | MUST | アップロード1枚あたり最大20MB。長辺上限は設けない |
| X07 | 画像フォーマット変換 | MUST | PNG/HEICをクライアントサイドでJPEGに変換（HEIC: heic2anyライブラリ） |
| X08 | サーバー側ファイルサイズ検証 | MUST | Presigned URLにContent-Length含め署名。confirm時にR2 HEADで実サイズ照合 |
| X09 | E2Eテスト | MUST | Playwrightで主要フローの動作検証 |
| X10 | サーバー側画像フォーマット検証 | MUST | confirm時にR2オブジェクトのマジックバイト（JPEG: FF D8 FF）を検証。画像以外のファイルをブロック |
| X11 | DL期限によるストレージ自動クリーンアップ | MUST | R13の期限切れ写真をCron Triggerで定期削除。R2オブジェクト + D1レコード + storage_used減算 |

## 4. 非機能要件

### 4.1 コスト

| 項目 | 目標 |
|---|---|
| 月額ランニングコスト（100受信者, 1TB） | ~$15/月 |
| ストレージ | Cloudflare R2（egress $0） |
| コンピュート | Cloudflare Workers Free/Paid |
| サムネイル生成 | クライアントサイドで$0 |

### 4.2 パフォーマンス

| 項目 | 目標 |
|---|---|
| サムネイル一覧表示 | <2秒（初回ロード） |
| 写真アップロード（10MB×1枚） | ネットワーク速度に依存。UIは即座にレスポンス |
| API応答 | <500ms（P95） |

### 4.3 可用性・スケーラビリティ

| 項目 | 値 |
|---|---|
| 初期想定ユーザー数 | 最大100受信者 |
| 同時送信者数 | 特に制限なし（Workers自動スケール） |
| 可用性目標 | Cloudflareプラットフォーム依存（99.9%+） |

### 4.4 セキュリティ

| 項目 | 対策 |
|---|---|
| 認証 | Firebase Auth IDトークン + Workers側で署名検証 |
| 認可 | 受信者は自分の写真のみアクセス可能 |
| ストレージアクセス | R2は全てプライベート。署名付きURLのみでアクセス |
| 送信者の認可 | セッションID（UUID v4）の知識に基づく |
| 画像の公開範囲 | 受信者のみ（プライベート） |
| NSFW検出 | 不要（初期リリース） |
| 通報機能 | 将来検討 |

### 4.5 対応プラットフォーム

| プラットフォーム | 方式 |
|---|---|
| Web（デスクトップ・モバイル） | PWA（Cloudflare Pages） |
| iOS | PWAとしてホーム画面に追加 |
| Android | PWAとしてインストール |

## 5. 画像仕様

| 項目 | 値 |
|---|---|
| 受付フォーマット | JPEG / PNG / HEIC（クライアントで全てJPEGに変換して保存） |
| 最大ファイルサイズ | 20MB/枚 |
| 想定解像度 | ~6000x4000（一眼レフ相当） |
| サムネイル | 長辺400px, JPEG品質0.7, クライアント生成 |
| EXIF | カメラモデルフィールドに送信者情報を埋め込み可能（元のカメラモデル情報は上書きされる） |
| 透かし | クライアントサイドでCanvas描画（位置・サイズ・透明度指定可） |

---

## 6. ストレージクォータ

| 項目 | 値 |
|---|---|
| 受信者あたりの上限（無料） | 10GB（一眼JPEG 15MB/枚で約680枚） |
| 1セッションあたりの枚数上限 | 100枚 |
| DL期限（デフォルト） | 30日（期限後に自動削除） |
| 有料プラン（Coming Soon） | 上限緩和・期限延長を予定 |
